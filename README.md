# smallhold

A single-binary, SQLite-backed Rust server for the fediverse. Speaks the Mastodon Client API well enough that Ivory, Phanpy, Elk, Tusky, and Tuba work against it without modification.

**One binary. One config file. One SQLite database. No Redis. No sidecars. No signup flow.**

Designed for a solo operator running a few dozen personas under one domain. Federation volume in the thousands of activities per day, not millions. Cold start under two seconds, steady-state memory under 150 MB.

---

## What this is not

- Not for public signups. Admin creates personas via CLI; there is no registration page.
- Not a drop-in Mastodon replacement. Missing: polls, lists, keyword filters, push notifications, full-text search, admin API, reports queue. Clients degrade gracefully on all of these.
- Not horizontally scalable. SQLite on one machine. At 30 personas this is fine.
- Not Kitsune. See [kitsune-soc/kitsune#681](https://github.com/kitsune-soc/kitsune/issues/681) for what happens when scope discipline fails.

---

## Requirements

- Linux x86_64 (musl static binary) or build from source on any Rust-supported platform
- A reverse proxy for TLS termination (Caddy recommended; nginx works)
- No other runtime dependencies

---

## Installation

Download the latest release binary:

```bash
curl -Lo smallhold https://github.com/yourname/smallhold/releases/latest/download/smallhold-x86_64-unknown-linux-musl
chmod +x smallhold
sudo mv smallhold /usr/local/bin/
```

Or build from source (requires Rust 1.75+):

```bash
cargo build --release
```

## Quick start

### 1. Initialize

```bash
smallhold init
```

Creates `config.toml` with a generated `secret_key`, initializes the SQLite database, and runs migrations. Edit `config.toml` before proceeding.

### 2. Set admin password

```bash
smallhold admin set-password
```

### 3. Create a persona

```bash
smallhold persona create writer --display-name="Your Name"
```

Each persona is a first-class ActivityPub actor with its own keypair, inbox, outbox, and followers collection. Mastodon clients see them as independent accounts.

### 4. Configure your reverse proxy

Caddy example (`Caddyfile`):

```
yourdomain.example {
    reverse_proxy 127.0.0.1:8080
    handle_path /media/* {
        root * /var/lib/smallhold/media
        file_server
        header Cache-Control "public, max-age=31536000, immutable"
    }
}
```

### 5. Start the server

```bash
smallhold serve
```

### 6. Connect a client

Open Ivory, Phanpy, Elk, Tusky, or Tuba. Enter your domain. Log in with the OAuth flow â€” it will ask for your admin password and which persona to authenticate as.

## Configuration

`config.toml`:

```toml
[server]
listen = "127.0.0.1:8080"
domain = "yourdomain.example"
secret_key = "<generated by smallhold init>"

[storage]
database_path = "/var/lib/smallhold/db.sqlite"
media_dir = "/var/lib/smallhold/media"

[federation]
user_agent = "smallhold/0.1 (+https://yourdomain.example)"
delivery_timeout_secs = 30
delivery_concurrency = 16
fetch_timeout_secs = 20
max_incoming_body_mb = 10
authorized_fetch = true

[limits]
max_post_chars = 5000
max_attachments = 4
max_media_mb = 40

[defaults]
default_visibility = "public"
default_sensitive = false
default_language = "en"

[branding]
site_title = "My Instance"
site_description = "A personal fediverse presence"
custom_css_path = "custom.css"  # optional, drop a CSS file to customize
```

### Custom theming

Drop a `custom.css` file and point `branding.custom_css_path` at it. The default styles use CSS custom properties, so overriding colors is one line:

```css
:root { --link: #e4002b; --bg: #fafafa; }
@media (prefers-color-scheme: dark) {
  :root { --link: #ff6b6b; --bg: #1a1a2e; }
}
```

Available variables: `--text`, `--muted`, `--bg`, `--card`, `--border`, `--link`.

## CLI reference

```
smallhold init                            # initialize DB, generate config skeleton
smallhold serve                           # start server

smallhold admin set-password
smallhold admin enable-totp

smallhold persona create <username> --display-name="..." [--locked] [--bot]
smallhold persona list
smallhold persona update <username> --bio="..." --display-name="..."
smallhold persona delete <username>       # sends Delete actor to followers, then removes
smallhold persona rotate-key <username>   # rotate RSA keypair, sends Update

smallhold token mint <username> --scopes=read,write,follow
smallhold token list
smallhold token revoke <token-id>

smallhold follow <username> <acct>        # e.g. smallhold follow writer gargron@mastodon.social
smallhold unfollow <username> <acct>

smallhold domain-block <domain> --severity=silence|suspend [--reject-media] [--reason="..."]
smallhold domain-unblock <domain>
smallhold domain-block-list

smallhold queue inspect                   # show pending deliveries
smallhold queue retry-dead                # reset permanently-failed rows

smallhold compat-test --target=mastodon.social --persona=writer
```

```
        polls delivery_queue table

```

IDs are 64-bit time-sortable snowflakes, serialized as strings. Mastodon client pagination (`since_id`, `max_id`, `min_id`) depends on this invariant.

## Pseudonymity note

All personas on one domain are attributable to one operator via the domain name, shared IP, and TLS fingerprint. For a persona that requires real separation (literary imprint, alt with different operational security), run a second smallhold instance on a separate VPS with a separate domain. Same binary, different config.
